# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

CertPilot-AutoDev is a job orchestration platform that connects Claude Code, Jira, Slack, and N8N for automated development workflows. It accepts tasks from multiple sources (Jira webhooks, Slack DMs, N8N schedulers), routes them through N8N for normalization, executes them via Claude Code in isolated jobs, and posts results back to Slack.

## Architecture

```
External Triggers (Jira/Slack/N8N/Schedules)
    → N8N Workflows (validation, normalization)
    → Claude Runner (Express server, port 3210)
    → Claude Code (subprocess execution)
    → Callback to N8N → Slack response / Follow-up dispatches
```

**Key Components:**
- `claude-runner/runner.js` - Express server that manages job queue, spawns Claude CLI, handles conversation memory, retry logic, and metrics
- `products/` - Per-product config (`product.json`) mapping workingDir to plugin directories
- `shared-skills/` - Platform-level shared skills (generic frameworks loaded for all products)
- `certpilot-plugin/` - CertPilot's plugin directory (agents, product-specific skills, commands, hooks)
- `<id>-plugin/` - Per-product plugin directories generated by the onboarding wizard
- `n8n/` - Docker Compose configuration for N8N + ngrok tunnel
- `*.json` (root) - N8N workflow definitions for Jira/Slack/callback handling

**Two Execution Modes:**
- **DELIVERY** (`/run`): Jira issue-driven, requires issueKey + workingDir
- **CHAT** (`/chat`): Conversational, maintains conversation memory per channel

## Commands

### Claude Runner
```bash
cd claude-runner
npm install

# Run server (RUNNER_SECRET required)
RUNNER_SECRET=your_secret node runner.js

# With all options
PORT=3210 HOST=127.0.0.1 RUNNER_SECRET=secret JOB_TIMEOUT_MINUTES=60 MAX_CONCURRENCY=1 node runner.js
```

### Docker (Recommended)

The entire stack (runner + dashboard + N8N + ngrok) can run via Docker Compose:

```bash
# Copy and configure environment
cp .env.example .env
# Edit .env with your RUNNER_SECRET and ANTHROPIC_API_KEY

# Build and start all services
docker compose build
docker compose up -d

# Check status
docker compose ps

# View runner logs
docker compose logs -f runner

# Stop everything
docker compose down
```

Services:
- **Runner**: http://localhost:3210 (API + health endpoint)
- **Dashboard**: http://localhost:3100 (Next.js UI)
- **N8N**: http://localhost:5678 (workflow automation)

The runner container uses `config.docker.json` with container-relative paths. Project directories are bind-mounted to `/projects/` and `ALLOWED_ROOTS` is set to `["/projects"]`.

### N8N (standalone)
```bash
cd n8n
docker-compose up -d    # Start N8N + ngrok
docker-compose down     # Stop services
```
Access N8N at http://localhost:5678 (admin/strongpassword)

### Dashboard (standalone dev)
```bash
cd dashboard
npm install
npm run dev             # http://localhost:3100
```

## API Quick Reference

All endpoints except `/health` require `x-runner-secret` header.

**Core:** `GET /health` · `POST /run` · `POST /chat` · `POST /agent` · `GET /agents` · `GET /jobs/:id` · `GET /jobs/:id/output`

**Pipelines:** `POST /pipeline` · `GET /pipelines/:id` · `POST /pipelines/:id/restart`

**Real-time:** `GET /events` (SSE) · `GET /jobs/:id/stream-events` · `GET /api/stats` · `GET /api/metrics`

**Operations:** `GET /api/failed-callbacks` · `POST /api/failed-callbacks/:id/replay`

**Inter-Agent:** `POST /internal/consult` (agents ask other agents inline, no queue)

## Key Patterns

### Multi-Product Plugin System
Two-layer skill architecture: **shared skills** provide generic frameworks (sales, security, architecture, QA, UX, BA, PM, reporting) while **product plugins** provide domain-specific context (brand, tech stack, Jira projects). The runner passes both via repeatable `--plugin-dir` flags:

```
claude --plugin-dir shared-skills --plugin-dir <product-plugin>
```

Each product gets its own plugin directory (`<id>-plugin/`) containing agents, skills, commands, hooks, company-brief, and MCP config. The runner resolves the product from `workingDir`, reads `pluginDir` from `products/<id>/product.json`, and merges with `shared-skills/`.

**Shared Skills** (`shared-skills/skills/`): sales, security, architecture, qa-testing, ux-design, ba-requirements, pm, reporting — product-agnostic, usable by all products.

**Onboarding**: Run `/onboard-product` to generate `products/<id>/product.json` + `<id>-plugin/` scaffold with product-specific agents.

**API**: `GET /api/products` lists all products; `POST /api/products/:id/reload` hot-reloads a config.

### Jira Integration
Use N8N Jira MCP tools for all Jira operations. Always call "get transitions" before transitioning issues.

### Subtask-Based Workflow (Primary)
Agents create `[CREATE-SUBTASKS]` blocks in output. Runner parses these and creates native Jira subtasks with dependency links and agent routing labels. Prefer this over legacy `[DISPATCH-ACTIONS]`.

> See [docs/claude/agents.md](docs/claude/agents.md) for full subtask format, agent labels, and delegation matrix.

### Pipeline Engine
Sequences SDLC phases as isolated, retryable jobs. Three types: `new-feature` (8 phases), `bug-fix` (3 phases), `security-fix` (4 phases). Labels `needs-requirements`/`needs-ux-design` auto-route to pipelines.

> See [docs/claude/pipelines.md](docs/claude/pipelines.md) for pipeline types, gate types, quality gates, and cross-phase coordination.

### Agent Teams
Team leads (engineer-planner, product-manager, sales-development) spawn teammates in-process for shared context. The engineering team: planner → implementer + ui-engineer + reviewer, all in one session.

> See [docs/claude/agents.md](docs/claude/agents.md) for Agent Teams configuration and when to use Teams vs Subtasks.

### Claude Execution
Spawns `claude --dangerously-skip-permissions --model <model-id> -p --output-format stream-json --verbose`. OAuth only — no API key. Pre-flight auth check via `ensureOAuthValid()`.

> See [docs/claude/reliability.md](docs/claude/reliability.md) for CLI flags, streaming details, and hooks system.

### N8N Workflows
Core workflows handle Jira webhooks, subtask events, runner callbacks, and scheduled automation (security scans, daily standup, sprint reports, sales pipeline, LinkedIn content).

> See [docs/claude/workflows.md](docs/claude/workflows.md) for all workflow files, Jira automation rules, and proactive PM capabilities.

### Marketing & Sales
Marketing content managed in Confluence MKTG space. LinkedIn engagement staged for human copy-paste. Sales pipeline automated via Attio CRM and weekly N8N schedules.

> See [docs/claude/integrations.md](docs/claude/integrations.md) for marketing routing, LinkedIn strategy, Attio CRM tools, and UI engineering handoff patterns.

### Reliability
State persisted to `~/claude-runner-logs/runner-state.json`. Callbacks retry 3x with exponential backoff. Budget circuit breaker at configurable daily/hourly limits.

> See [docs/claude/reliability.md](docs/claude/reliability.md) for full reliability details, hooks, and the CertPilot plugin structure.
